# AI 协作开发工作流

## 1. 目标

这份文档约定《苍纹牌阵》后续如何用 AI 稳定推进系统、关卡、UI 与剧情内容，避免一边聊天一边散改代码，导致方向漂移或返工。

核心原则：

- 先文档
- 再概念图
- 再小规格
- 再实现
- 最后统一验证

## 2. 标准迭代闭环

每个新任务都按以下顺序执行：

1. 写小规格
2. 出概念图或视觉参考
3. 明确实现边界
4. 让 AI 实现
5. 跑检查脚本
6. 在预览场景或真实流程中验证
7. 记录结果与下一步

不允许直接跳到“帮我把这个做了”而没有规格和验收。

## 3. 哪些任务必须先出图

以下任务必须先做概念图或视觉需求说明：

- 新战斗 HUD
- 新棋盘表现层
- 章节地图
- 营地界面
- 奖励、转职、结算
- 新角色立绘气质与配色
- 任何会明显改变画面信息层级的新系统

图不一定一步到位，但必须先有方向图，再进入正式实现。

## 4. Hastur 插件使用边界

项目内已启用 `HasturOperationGD` 插件。它是正式工具链的一部分。

推荐用 Hastur 做：

- 查询当前场景树
- 创建或重命名节点
- 调整锚点、层级、导出属性
- 检查节点路径与引用关系
- 保存场景结构

不推荐把 Hastur 当作视觉微调替代品。细节拖拽、边距和视觉观感，仍优先在 Godot 编辑器里人工确认。

## 5. AI 任务模板

后续给 AI 的任务，默认按这个模板写：

### 目标

一句话说明这轮要解决什么问题。

### 改动范围

列出允许修改的系统、文档、场景或数据文件。

### 不可触碰项

写明这轮不能顺手改什么，避免扩散。

### 概念图输入

如果任务涉及视觉，写明参考图、风格锚点、不可偏离项。

### 验收

必须写出：

- 功能是否生效
- 需要跑哪些脚本
- 需要在什么场景或流程里验证

### 输出物

例如：

- 代码改动
- JSON 调整
- 新文档
- 概念图需求稿
- 测试说明

## 6. 四类常见任务建议

### 新系统

- 先写规则白皮书
- 再写预览需求
- 再实现最小闭环
- 最后补 UI 文案与反馈

### 新关卡

- 先写关卡目标与教学重点
- 再画 TileMap
- 再补 JSON 出生点、目标、招募与教学
- 最后跑校验与关卡预览

### 新角色

- 先定职责、被动、剧情钩子
- 再写技能与关卡出场
- 再补美术需求
- 最后接入战斗与营地

### 新 UI

- 先写信息层级与交互目标
- 再出概念图
- 再用 Hastur + UiPreview 调结构
- 最后回 Main 联调真实流程

## 7. 固定验证流程

所有正式实现后至少执行：

```powershell
.\tools\check_all.ps1
```

UI 或流程相关任务额外执行：

- 打开 `scenes/UiPreview.tscn`
- 打开 `scenes/Main.tscn`
- 检查状态切换、按钮行为、遮挡关系

关卡相关任务额外执行：

- 打开 `scenes/LevelPreview.tscn`
- 检查地形、出生点、教学高亮、目标格

## 8. 长任务维护

长任务期间默认定期使用 `keep-codex-fast` 做 report-only 检查。

原则：

- 只先做报告
- 不自动清理
- 不自动归档活跃线程
- 重要长线程在清理前先写 handoff

后续开发默认优先维护这些文档：

- `docs/game_direction.md`
